## Architecture

```
Browser A ──┐                          ┌── Browser B
            │  WebSocket (Nakama RT)   │
            └──────► Nakama Server ◄──┘
                          │
                    (PostgreSQL)
                          │
                   Match State (RAM)
                   Leaderboard (DB)
```

**Server-authoritative**: every game action (move, ready, timeout) is validated and applied exclusively on the Nakama server. Clients only send intent; they never mutate state directly.

## Features

### Core
- **Server-authoritative game logic** – all moves validated on the server; clients cannot cheat
- **Real-time state broadcast** – every match-loop tick pushes the full board to all connected clients
- **Quick Match** – automatic 1-vs-1 matchmaking via Nakama's matchmaker
- **Private Rooms** – create a room and share the Room ID with a friend
- **Graceful disconnect handling** – opponent disconnect awards the remaining player a win
- **Concurrent game isolation** – each match runs in its own server-side state object

### Bonus
- **Leaderboard** – global win ranking, persisted in PostgreSQL, top-10 via RPC
- **Timed mode** – 30-second turn timer with auto-forfeit; both players must vote for timed mode
- **Countdown UI** – live timer in the client with urgent animation when < 10 s remain

## Local Development

### Prerequisites
- Docker Desktop (or Docker Engine + Compose)
- Node.js 20+

### Steps

```bash
# 1. Clone the repo
git clone <your-repo-url>
cd tictactoe

# 2. Build the Nakama TypeScript module
cd nakama
npm install
npm run build          # outputs nakama/data/modules/tictactoe.js
cd ..

# 3. Copy frontend env file
cp frontend/.env.example frontend/.env.local

# 4. Start the full stack
docker compose up --build

# Services available:
#   Nakama API  →  http://localhost:7350
#   Nakama Console → http://localhost:7349  (admin/adminpassword)
#   React app   →  http://localhost:5173
```

> **Hot reload**: the frontend container mounts `./frontend` as a volume, so Vite HMR works as usual.

## AWS Deployment

### Prerequisites
- AWS CLI configured (`aws configure`)
- Docker running locally
- An AWS account

### Step-by-step

#### 1. Build and push the frontend image

```bash
# Authenticate to ECR (replace ACCOUNT_ID and REGION)
aws ecr create-repository --repository-name tictactoe-frontend --region us-east-1
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  <ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com

# Build the Nakama JS module first
cd nakama && npm ci && npm run build && cd ..

# Build + push frontend (set NAKAMA_LB_DNS to the Nakama ALB DNS name after the CF stack creates it)
docker build \
  --build-arg VITE_NAKAMA_HOST=<NAKAMA_ALB_DNS> \
  --build-arg VITE_NAKAMA_PORT=7350 \
  --build-arg VITE_NAKAMA_USE_SSL=false \
  --build-arg VITE_NAKAMA_SERVER_KEY=defaultsocketkey \
  -t <ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/tictactoe-frontend:latest \
  ./frontend

docker push <ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/tictactoe-frontend:latest
```

#### 2. Build and push the custom Nakama image

```bash
aws ecr create-repository --repository-name tictactoe-nakama --region us-east-1

docker build \
  -f nakama/Dockerfile.nakama \
  -t <ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/tictactoe-nakama:latest \
  .

docker push <ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/tictactoe-nakama:latest
```

#### 3. Deploy CloudFormation stack

```bash
aws cloudformation deploy \
  --stack-name tictactoe \
  --template-file deploy/aws/cloudformation.yml \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameter-overrides \
    ProjectName=tictactoe \
    NakamaImage=<ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/tictactoe-nakama:latest \
    FrontendImage=<ACCOUNT_ID>.dkr.ecr.us-east-1.amazonaws.com/tictactoe-frontend:latest \
    DBPassword=<STRONG_PASSWORD> \
    NakamaServerKey=<YOUR_SERVER_KEY> \
  --region us-east-1
```

#### 4. Get endpoints

```bash
aws cloudformation describe-stacks \
  --stack-name tictactoe \
  --query "Stacks[0].Outputs" \
  --output table
```

| Output         | Description                          |
|----------------|--------------------------------------|
| `FrontendURL`  | Public game URL                      |
| `NakamaEndpoint` | Nakama API host (for client config) |

## Nakama API Reference

All game communication goes over the Nakama real-time socket. REST RPCs are also available.

### Op-codes (WebSocket match data)

| Code | Direction          | Payload                         | Description              |
|------|--------------------|---------------------------------|--------------------------|
| `1`  | Server → Client    | `ServerState` JSON              | Full game state snapshot |
| `2`  | Client → Server    | `{ "index": 0-8 }`             | Make a move              |
| `3`  | Client → Server    | `{ "timed": true\|false }`     | Signal ready + mode vote |
| `4`  | Bidirectional      | _(empty)_                      | Keep-alive ping          |

### RPC Endpoints

| RPC ID            | Method | Payload                      | Response                         |
|-------------------|--------|------------------------------|----------------------------------|
| `create_match`    | POST   | `{ "timed": true\|false }`  | `{ "matchId": "..." }`           |
| `get_leaderboard` | POST   | `{}`                         | `{ "records": [...] }`           |

### ServerState Schema

```jsonc
{
  "board":       [null, "X", null, "O", null, null, null, null, null],
  "currentTurn": "<sessionId of player whose turn it is>",
  "phase":       "lobby" | "playing" | "finished",
  "winner":      "<sessionId>" | "draw" | null,
  "timedMode":   false,
  "turnStart":   1712300000000,   // epoch ms
  "turnTimeSec": 30,
  "symbols":     { "<sessionId>": "X", "<sessionId2>": "O" }
}
```

## Testing Multiplayer

### Option A – Two browser tabs (same machine)

1. `docker compose up` (with the module built)
2. Open `http://localhost:5173` in Tab 1 → enter a username → **Create Private Room**
3. Copy the Room ID shown → open `http://localhost:5173` in Tab 2 (incognito) → enter a different username → paste Room ID → **Join**
4. Both players appear in the lobby → select a mode → game starts

### Option B – Quick Match across two machines

Both machines must point to the same Nakama instance (update `.env.local`).  
Click **Quick Match** on both within a ~10-second window; Nakama's matchmaker pairs them automatically.

### Option C – Nakama Console

1. Open `http://localhost:7349` (local) or the console URL (AWS)  
2. Login: `admin` / `adminpassword`  
3. Navigate to **Matches** to inspect live game state, connected users, and match labels

## Design Decisions

| Decision | Rationale |
|---|---|
| **TypeScript module (not Lua)** | Better type safety; esbuild compiles to single JS file Nakama can load |
| **Match loop broadcasts every tick** | Simple and reliable; state is always fresh on the client |
| **`null` return from matchLoop** | Cleanly terminates the match server-side after the game ends |
| **Leaderboard: INCREMENTAL operator** | Allows `subscore` to accumulate wins without a read-modify-write cycle |
| **Timed mode consensus** | Both players must vote `timed=true`; minority vote → classic mode (avoids unfair advantage) |
| **Device-id auth** | Zero-friction login; stable identity across sessions via `localStorage` |
| **Fargate (serverless containers)** | No EC2 instance management; scales to zero between games |
| **CloudFormation IaC** | Fully reproducible infrastructure; single command deploy |
